## 一、项目概况

### 1.1 服务架构

```
浏览器 (React :3000)
  │  Vite proxy: /api → localhost:80
  ▼
Traefik (:80)                                    ← API网关
  │  middleware: cors → rate-limit → security → retry
  ├── /api/v1/users/*        → user-service:8080           (Go/go-zero)
  ├── /api/v1/cases/*        → content-service:8081         (Go/go-zero)
  ├── /api/v1/works/*        → content-service:8081
  ├── /api/v1/comments/*     → script-service:8000          (Python/FastAPI)
  ├── /api/v1/scripts/*      → script-service:8000
  ├── /api/v1/storyboard/*   → storyboard-service:8001      (Python/FastAPI)
  ├── /api/v1/llmhua/*       → llmhua-service:8002          (Python/FastAPI)
  ├── /api/v1/videos/*       → video-service:8000           (Python/FastAPI+Celery)
  ├── /api/v1/final-cut/*    → final-cut-service:8085       (Go/go-zero/gRPC)
  └── /api/v1/recommendations/* → recommendation-service:8004 (Python/FastAPI)

基础设施:
  MySQL 8.0 (:3306)     Redis 7 (:6379)    RabbitMQ 3 (:5672)
  MinIO (:9000)         Kafka 3.9 (:9092)  Elasticsearch 8.17 (:9200)
  Consul 1.20 (:8500)   Prometheus (:9090) Grafana (:3001)
  Jaeger (:16686)
```

### 1.2 技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 前端 | React 18 + TypeScript + Vite + Ant Design 5 | Redux Toolkit, React Router 6 |
| Go服务 | go-zero v1.10 + gRPC | DBResolver读写分离, Consul服务发现 |
| Python服务 | FastAPI + Uvicorn + Gunicorn | LangChain + LangGraph + FAISS |
| LLM | DeepSeek/OpenAI/Anthropic/vLLM | 多模型路由, prompt caching |
| 图像/视频 | Seedream 4.5 / Seedance 2.0 | 火山引擎Ark API |
| Embedding | bge-large-zh-v1.5 / TEI | 1.3GB CPU / GPU服务化 |
| ML | PyTorch Wide&Deep + LinUCB Bandit | CTR预估 + 在线学习 |

## 二、核心API调用全链路

### 2.1 从大纲生成剧本 — 同步请求

**请求**: `POST /api/v1/scripts/generate/from-outline-sync`

**业务场景**: 用户输入简短创意（几十到几百字），AI 扩展为完整短剧剧本（含角色设定+分集大纲+分镜表）。

**请求体**:
```json
{
  "title": "重生之都市修仙",
  "outline": "修真者重生到现代都市，开启逆天修行之路，却发现这个世界的秘密远超想象...",
  "theme": "奇幻",
  "length": "短篇",
  "style": "古装风格",
  "stream": false
}
```

**完整处理流程**:

```
T+0ms   浏览器 POST → Traefik(:80) → script-service:8000/.../from-outline-sync

T+2ms   Traefik 中间件链: cors → rate-limit-ai(50/s) → security-headers → retry

T+5ms   script-service 接收, FastAPI → PrometheusMiddleware → RateLimitMiddleware
          ├── 限流检查: Redis INCR ratelimit:ai_generate:{user}:{minute}
          │    超过10次/分钟 → 429 + Retry-After
          └── endpoint: generate_from_outline_sync()

T+10ms  ┌── 参数处理 ─────────────────────────────────────┐
        │ 1. 验证 title/outline 非空                      │
        │ 2. 获取用户偏好画像:                              │
        │    get_user_preference_service()                  │
        │    → 查询 scripts 表 (用户历史风格/题材/长度)     │
        │    → 构建 user_context: "用户偏好风格:古装,偏好题材:奇幻"│
        │ 3. 确定目标集数:                                  │
        │    length="短篇" → target_eps=5                  │
        │    (短片5集, 中篇8集, 长篇12集, 最多50)          │
        │ 4. 解析大纲中的显式集数要求:                      │
        │    regex提取 "20集" "二十集" → target_eps=20     │
        │ 5. 创建 Novel2ScriptV2Service:                   │
        │    llm = ResilientLLM(deepseek-chat)             │
        │    config = app_settings (chunk_size=4096...)     │
        └──────────────────────────────────────────────────┘

T+15ms  ┌── V2 Pipeline (大纲模式) ────────────────────────┐
        │                                                   │
        │ Stage 1: 知识库 (T+15ms ~ T+200ms)               │
        │   大纲很短(几十到几百字), FAISS只分1-2个chunk     │
        │   但大纲模式下RAG基本不用(章节内容=大纲全文)        │
        │                                                   │
        │ Stage 2: 章节检测 (T+200ms)                       │
        │   大纲无"第X章"标记 → 检测到1章("全文")           │
        │   len(chapters)==1 && target_eps(5)>1             │
        │   → 大纲扩展模式: 创建5个虚拟章节                  │
        │   每个虚拟章节都包含完整大纲+序数提示               │
        │   chapter.episode_hint = "这是第1集,共5集,请创作本集"│
        │                                                   │
        │ Stage 3: 角色提取 (T+200ms ~ T+5s)               │
        │   LLM从大纲提取潜在角色/关系/场景/道具              │
        │   (大纲短, 提取很快)                               │
        │                                                   │
        │ Stage 3b: 故事框架开发 ★ 大纲核心步骤 (T+5s ~ T+20s)│
        │   chapters<5 → 执行! (完整小说则跳过)             │
        │   LLM将简短大纲扩展为完整故事框架:                 │
        │   【故事梗概】200-500字                            │
        │   【角色设定】每个角色: 性格+动机+弧线+关系        │
        │   【分集大纲】5集中每集的: 标题+冲突+关键事件+钩子  │
        │   【核心场景】5-10个地点                           │
        │   【核心道具】3-5个关键道具                        │
        │   → 这些内容注入到后续每章的prompt中                │
        │                                                   │
        │ Stage 3c: 章节摘要 (跳过)                          │
        │   大纲只有1章→摘要无意义                           │
        │                                                   │
        │ Stage 4: 逐集剧本生成 (T+20s ~ T+100s)            │
        │   asyncio.Semaphore(3) 并发, 5集并行处理          │
        │   每集调用LLM, prompt包含:                         │
        │     global_info (角色关系图谱)                     │
        │     story_framework (分集大纲中的该集内容)          │
        │     graphrag_context (如有知识图谱上下文)           │
        │     user_preferences (用户偏好风格/题材)            │
        │     chapter_content = 大纲原文                     │
        │     episode_hint = "这是第N集,共5集"               │
        │     style_rule = 古风台词风格                      │
        │   输出: 场景编号+类型+地点+角色+道具+分镜表+剧本正文│
        │                                                   │
        │ Stage 5: 主体提取 (T+100s ~ T+110s)               │
        │   从生成的5集剧本中提取角色/地点/道具               │
        │                                                   │
        │ Stage 6: 质量保障 (T+110s ~ T+120s)               │
        │   内容安全检查 (敏感词->reject)                    │
        │   QualityJudge 4维度评分                          │
        │   verdict=retry → 注入反馈自动重试                 │
        └──────────────────────────────────────────────────┘

T+125s  组装响应 → 5集EpisodeItem[] + ShotEpisode[]
T+130s  用量上报: tokens(真实从LLM response提取) × DeepSeek定价
T+135s  返回 JSON 给浏览器
```

**返回结果**:
```json
{
  "script_id": 0,
  "title": "重生之都市修仙",
  "episodes": [
    {"episode_number": 1, "title": "第一集", "content": "第一集\n\n【场景编号】SCENE-001\n..."},
    {"episode_number": 2, "title": "第二集", "content": "..."},
    ...
  ],
  "total_episodes": 5,
  "storyboard": [{"shot_number":1, "camera_type":"全景", ...}, ...],
  "quality_report": {"total_score": 82, "verdict": "pass", ...},
  "safety_report": {"passed": true, "score": 100}
}
```

### 2.2 从小说生成剧本 — 异步请求

**请求**: `POST /api/v1/scripts/generate/from-novel`

**业务场景**: 用户上传完整小说文本（几万~几十万字），AI 识别章节后逐章改编为剧本。

**与大纲模式的关键差异**:
- 大纲模式：内容短，重点在 LLM 扩展创造
- 小说模式：内容长，重点在 RAG 检索+忠实改编

```
T+0ms   POST → script-service:8000
        返回: {"task_id":"uuid","status":"processing"}
        BackgroundTasks 触发异步处理

T+5ms   ┌── V2 Pipeline (小说模式) ────────────────────────┐
        │                                                   │
        │ Stage 1: FAISS 知识库 (T+5ms ~ T+30s)             │
        │   小说分块 (chunk=4096, overlap=512)              │
        │   bge-large-zh-v1.5 embedding 全部chunks          │
        │   磁盘缓存: faiss_cache/{md5}.faiss               │
        │   下次相同小说直接加载缓存                          │
        │                                                   │
        │ Stage 2: 章节检测 (T+30s)                          │
        │   regex: "第X回/章/节" or "Chapter N"             │
        │   例: 50章 → 50个独立章节片段                      │
        │   chapters>=5 → 完整小说模式                       │
        │                                                   │
        │ Stage 3: 角色关系图谱 (T+30s ~ T+40s)             │
        │   LLM提取全局角色/关系/场景/道具                   │
        │   输入截断: novel[:80000] (长篇小说只看前8万字)     │
        │                                                   │
        │ Stage 3b: 故事框架 (跳过!)                         │
        │   chapters>=5 → 完整小说, 不需要AI扩展             │
        │   直接用小说原文的章节结构                          │
        │                                                   │
        │ Stage 3c: 章节摘要 (T+40s ~ T+70s) ★ 核心差异     │
        │   逐章串行调用LLM (30s/chapter)                   │
        │   每章生成100-200字摘要                            │
        │   构建跨章上下文:                                   │
        │     第5章获得第2-4章摘要 (最近3章) + 第6章预告     │
        │   保证50章长篇的角色/伏笔/事件连贯性                │
        │                                                   │
        │ Stage 4: 逐章剧本生成 (T+70s ~ T+300s)            │
        │   每章:                                            │
        │     ├── RAG检索: FAISS.top_k=5                   │
        │     │   语义query: "角色:张三 李四 场景:京城 王府" │
        │     │   从全文50章中检索最相关的前文片段            │
        │     ├── 智能截断: 首6000+尾2000 (保留首尾剧情)    │
        │     ├── 注入跨章上下文 (前情提要)                  │
        │     └── LLM生成 (180s timeout)                   │
        │   asyncio.Semaphore(3) 并行, 50章→~17批           │
        │                                                   │
        │ Stage 5-6: 同上 (主体提取+质量评审)                │
        └──────────────────────────────────────────────────┘

T+310s  TaskStore.update(task_id, {status:"completed", result:{...}})
        前端轮询 GET /{task_id}/status → completed → 获取结果
```

### 2.4 推荐请求全链路

**请求**: `GET /api/v1/recommendations/recommend?user_id=1&limit=6`

```
T+0ms   Traefik → recommendation-service:8004

T+2ms   RecommendationPipeline.run():
          ├── Layer 1: 5路并行召回 (asyncio.gather)
          │     ├── _collaborative(user_id):
          │     │     SELECT cases FROM users_who_interacted_with_same_cases
          │     │     → 50 items, recall_score=0.9, source="cf"
          │     ├── _content_based(user_id):
          │     │     提取用户历史标签 → FIND_IN_SET匹配 → 60 items, source="content"
          │     ├── _hot():
          │     │     (like_count*3+view_count)*time_decay → 50 items, source="hot"
          │     ├── _author_based(user_id):
          │     │     用户关注作者的新作品 → 30 items, source="author"
          │     └── _search(query): (如果存在)
          │           SQL LIKE 匹配 → 30 items, source="search"
          │     → 合并去重, 按recall_score排序, 截取前200
          │
          ├── Layer 2: 过滤
          │     ├── 查询 user_case_interactions (已看的case_id)
          │     └── 去掉已看 + 去重 → ~150 items
          │
          ├── _get_user_profile(user_id):
          │     SELECT 用户历史行为统计 (view/like/share counts)
          │     SELECT 用户top-5标签
          │     SELECT 用户最喜欢的genre
          │     SELECT 用户top-3作者
          │     SELECT 用户最近活跃时间+活跃时段
          │
          ├── Layer 3: Wide&Deep 排序
          │     ├── 提取13维特征 (10连续+2类别+1二元)
          │     │     user_view_count, user_like_count, tag_diversity,
          │     │     item_view_count, item_like_count, item_age_days,
          │     │     tag_match_count, recall_source (embedding), genre (embedding),
          │     │     genre_match, author_match, hour_of_day
          │     ├── FeaturePreprocessor: Z-score归一化
          │     ├── WideAndDeep.forward():
          │     │     Deep: (13→256→128→64) + BatchNorm + ReLU + Dropout
          │     │     Wide: Linear(13→1) + 类别embedding
          │     │     Final: sigmoid(Wide⊕Deep)
          │     ├── 模型不可用 → 降级为加权公式:
          │     │     ctr = 0.3*like_rate + 0.1*view_rate
          │     │         + 0.2*recall_score*(0.5+bandit_weight)
          │     │         + 0.05*random (exploration noise)
          │     └── Bandit: LinUCB per-user per-source 在线调整权重
          │
          └── Layer 4: MMR重排
                λ=0.7, Jaccard tag相似度
                贪心选择 top-6, 保证多样性

T+50ms  返回: {items:[...], reason:"personalized (cf,content,hot)", total:6}
```

## 三、异步任务处理流程

### 3.1 任务生命周期

```
                   ┌──────────────────────────────┐
                   │     POST /xxx/generate        │
                   │     → 返回 task_id            │
                   └──────────┬───────────────────┘
                              │
              ┌───────────────┼───────────────┐
              │               │               │
         MemoryQueue    KafkaQueue      BackgroundTasks
         (开发环境)     (生产环境)       (Celery worker)
              │               │               │
              ▼               ▼               ▼
         ┌──────────────────────────────────────┐
         │  Worker消费 → 执行AI生成 → 更新状态    │
         │  TaskStore (Redis):                   │
         │    {status, progress, result, error,  │
         │     start_time, end_time}             │
         └──────────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
         前端轮询                          SSE推送
    GET /{id}/status                  event: progress
    (2s间隔)                          event: done
```

### 3.2 Kafka Topic 设计

```
ai-tasks-p1-realtime   6 partitions   保留1h   实时AI请求 (延迟<2s)
ai-tasks-p2-video      12 partitions  保留2h   视频生成 (延迟<5min)
ai-tasks-p3-batch      24 partitions  保留24h  批量离线
ai-tasks-dlq           3 partitions   保留7天   死信队列
```

### 3.3 KEDA 自动扩缩

```
Kafka lag > 5 messages (ai-tasks-script topic)
  → KEDA triggers script-service Deployment
  → scale from 1 → up to 30 pods
  → 每15s检查一次lag
  → lag降到0后, 冷却5分钟再缩容

Cron trigger:
  → 每天凌晨2-6点, 维持3个pod跑离线预生成
```

## 四、多模型容错与路由

### 4.1 ModelRouter 初始化

```
create_llm_client(prefer="deepseek")
  │
  ├── _try_create_deepseek()
  │     ├── 读 DEEPSEEK_API_KEY (env)
  │     ├── 创建 httpx.AsyncClient(base_url="https://api.deepseek.com")
  │     ├── ChatOpenAI(model="deepseek-chat", streaming=True, max_tokens=16000)
  │     └── ✓ → 返回 ResilientLLM(llm, provider="deepseek")
  │
  ├── (fallback) _try_create_openai()
  │     └── ChatOpenAI(model="gpt-4o", streaming=True)
  │
  ├── (fallback) _try_create_anthropic()
  │     ├── 优先: ANTHROPIC_API_BASE (OpenAI兼容代理)
  │     └── 备选: langchain_anthropic ChatAnthropic (直连)
  │
  └── (fallback) _try_create_vllm()
        ├── 读 VLLM_API_BASE (env, 如 http://vllm-server:8000/v1)
        ├── ChatOpenAI(model=Qwen/Qwen2.5-7B-Instruct)
        └── 自托管GPU推理, 零API成本

返回: ResilientLLM (包装器, 运行时故障转移)
```

### 4.2 ResilientLLM 运行时切换

```
async def ainvoke(messages):
    breaker = get_circuit_breaker("llm-deepseek")  # 熔断器

    for attempt in range(3):
        try:
            result = await breaker.call(self._llm.ainvoke, messages)
            self._mark_healthy()
            return result
        except CircuitOpenError:            # 熔断器已打开
            logger.warning("breaker OPEN → switching provider")
            self._mark_unhealthy()
            self._switch_provider()           # deepseek → openai
            breaker = get_circuit_breaker("llm-openai")
        except Exception as e:
            if is_retryable(429/5xx):
                self._mark_unhealthy()
                self._switch_provider()
            else:
                raise

    raise RuntimeError("all providers exhausted")

故障场景:
  DeepSeek 频繁429 → 5次错误/30s → CircuitBreaker OPEN
  → 自动切到 OpenAI → OpenAI 正常 → 继续服务
  → 30s后探测 DeepSeek → 恢复正常 → 切回
```

TieredModelRouter.get_llm(tier):

## 五、缓存体系

### 5.1 三级缓存

```
请求到达
  │
  ├── L1: 内存LRU (cache_layers.py)
  │     OrderedDict, 1000 entries, 60s TTL
  │     O(1) 查找, <0.1ms
  │     hit → 返回 (70%命中率)
  │
  ├── L2: Redis (cache_service.py)
  │     MD5(request)→JSON, TTL 2h
  │     网络往返 ~1ms
  │     hit → 返回 + 晋升到L1 (20%命中率)
  │
  └── L3: 语义缓存 (semantic_cache.py)
        FAISS 向量索引, bge-large-zh-v1.5
        余弦相似度>0.92 → 返回缓存
        命中率: 60-80% (热点请求)
        例: "重生修仙"和"穿越修仙"相似度0.94 → 复用结果
```

### 5.2 缓存键设计

```python
# Redis 缓存键
storyboard:md5(title+theme+style+scene_count+script_hash[:16])
shots:md5(title+theme+style+episode_count+episode_contents_hash[:8])
script:md5(title+theme+length+style+setting+content_hash[:12])

# 语义缓存键
FAISS: {request_text_embedding} → {similarity_score} → cached_result
```

### 5.3 缓存失效

```
- 流式请求 (stream=true): 跳过所有缓存, 直接调LLM
- TTL自动过期: 剧本2h, 分镜2h, 工作流3h
- 手动清除: POST /api/v1/scripts/clear-cache → Redis FLUSHDB
- 语义缓存: evict oldest when > 10000 entries
```

## 六、SSE流式输出

### 6.1 协议

```
SSE (Server-Sent Events)
Content-Type: text/event-stream
Headers: Cache-Control:no-cache, Connection:keep-alive, X-Accel-Buffering:no

事件类型:
  event: token     LLM token级流式输出
  event: stage     多阶段流水线进度
  event: progress  图片/视频生成进度
  event: done      完成(含最终结果)
  event: error     错误(含code)

心跳: : heartbeat\n\n (每15s, 仅progress类型使用)
```

### 6.2 启用方式

```bash
# 单端点双模: stream参数 (默认false, 向后兼容)
POST /api/v1/scripts/generate/from-outline-sync
{"stream":true, "title":"...", ...}

# 全局关闭: 环境变量
SSE_STREAMING_ENABLED=false  # 即使传stream=true也不启用
```

### 6.3 服务端实现

```python
# 端点层 (script.py)
use_streaming = getattr(request, "stream", False) and settings.SSE_STREAMING_ENABLED
if use_streaming:
    return StreamingResponse(event_generator(), media_type="text/event-stream")

# 服务层 (ai_service.py)
async def _stream_llm_response(system, human):
    messages = [SystemMessage(system), HumanMessage(human)]
    async for sse_event in stream_tokens_from_llm(self.llm, messages):
        yield sse_event  # 每token一个event: token

# 工具层 (sse.py)
async def stream_tokens_from_llm(llm, messages):
    async for chunk in llm.astream(messages):
        yield format_sse_event({"text": token, "accumulated": full}, event="token")
    yield format_sse_event({"text": "", "complete": True}, event="token")
```

### 11.1 剧本质量自动评审

QualityJudge.judge_script(content):
